# Feature: Delete

> 상위 문서: [[../Plan]]

## 목적

id로 지정한 레코드를 사용자 확인을 거쳐 안전하게 삭제하고 `data.json`에 반영한다.
"안전하게"는 다음 두 가지를 의미한다: (1) 삭제 전 대상 확인 및 명시적 승인 절차,
(2) 삭제 직전 파일 백업으로 실수 복구 가능성 확보.

## 관련 API

- `PersonStore::findById(int id) -> const json::Value*` — 삭제 대상 조회 및 존재 확인
- `PersonStore::removeById(int id) -> bool` — 실제 삭제 수행, 성공 여부 반환

## 사용자 흐름

```
-- Delete --
삭제할 id 입력: 2

삭제 대상 데이터:
{
  "id": 2,
  "name": "김철수",
  "age": 22,
  "isStudent": true,
  "skills": []
}

정말 삭제하시겠습니까? (y/n): y

삭제 완료 (id=2)
```

취소 시:

```
정말 삭제하시겠습니까? (y/n): n
삭제를 취소했습니다.
```

## 처리 절차

1. id 입력받아 `findById`로 존재 확인. 없으면 "id=.. 에 해당하는 데이터가 없습니다." 안내 후 메뉴로 복귀
2. 대상 레코드 전체를 `dump(2)`로 출력해 사용자가 삭제 대상을 명확히 인지하게 함
3. `y/n` 확인 프롬프트 — `y`(대소문자 무관)가 아니면 전부 취소로 간주하고 메뉴로 복귀 (파일 변경 없음)
4. 확인 시:
   a. 삭제 직전 현재 `data.json`을 `data.json.bak`으로 복사 (덮어쓰기, 최근 1회분만 유지)
   b. `PersonStore::removeById(id)` 호출로 메모리에서 레코드 제거
   c. `save()` 호출로 `data.json`에 즉시 반영
5. 삭제 완료 메시지 출력 (`삭제 완료 (id=2)`)

## 안전장치 상세

- **확인 절차 없이는 삭제 API를 호출하지 않는다.** CLI 계층에서 `y` 확인 전에는
  `PersonStore::removeById`를 호출하지 않도록 한다 (Store 자체는 확인 로직을 모른다 — 책임 분리).
- **백업 파일**: `data.json.bak`은 매 삭제마다 직전 상태로 덮어써지는 단일 백업이다.
  여러 세대를 보관하지 않는다 (요구사항 범위를 벗어나는 과설계 방지).
- **id 재사용 금지**: 삭제된 id는 이후 Create에서 다시 부여되지 않는다
  (`nextId()`는 현재 남아있는 레코드의 최대 id + 1을 사용하므로, 삭제 후에도 다른 레코드의 id와
  충돌하지 않는 것만 보장하면 되고 굳이 "삭제된 적 있는 id" 이력을 별도로 관리하지 않는다).

## 에지 케이스

- 존재하지 않는 id 삭제 시도: 안내 메시지만 출력, 파일/백업 변경 없음
- 마지막 남은 레코드 삭제: 삭제 후 `data.json`은 빈 배열 `[]`이 되며 이후 Read는 "저장된 데이터가 없습니다." 출력
- 확인 프롬프트에서 `y`/`n` 이외 값 입력: `n`과 동일하게 취소로 처리 (모호한 입력은 안전한 쪽으로)
- 백업 파일 쓰기 실패(디스크 권한 등): 백업 실패 시 삭제를 진행하지 않고 오류를 안내
  (백업 없는 삭제는 "안전한 삭제" 요구사항 위반이므로 허용하지 않음)

## 테스트 체크리스트

- [ ] 존재하는 id 삭제 확인(y) 시 레코드가 제거되고 파일에 반영되는지
- [ ] 확인 프롬프트에서 n 입력 시 레코드가 삭제되지 않는지
- [ ] 존재하지 않는 id 삭제 시도 시 파일이 변경되지 않는지
- [ ] 삭제 시 `data.json.bak`이 삭제 직전 상태로 생성/갱신되는지
- [ ] 마지막 레코드 삭제 후 `data.json`이 빈 배열로 남는지
- [ ] 삭제 후 동일 id로 새 레코드를 Create해도 id가 재사용되지 않는지 (다음 id는 계속 증가)
